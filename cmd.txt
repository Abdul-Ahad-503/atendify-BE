========================
DATABASE MODELS (MONGOOSE)
========================

1. User Schema:
   - role: enum ['student', 'teacher', 'admin']
   - email, password (hashed)
   - name
   - student_id (if student)
   - employee_id (if teacher)
   - course, semester (if student)
   - department
   - subjects_taught (if teacher)
   - is_active
   - created_at, updated_at

2. Subject Schema:
   - name, code
   - department
   - semester
   - teacher_id (ref: User)
   - enrolled_students (array of student IDs)

3. Room Schema:
   - name, number
   - building, floor
   - latitude, longitude
   - geofence_radius (meters)

4. Timetable Schema:
   - subject_id (ref: Subject)
   - teacher_id (ref: User)
   - room_id (ref: Room)
   - course, semester (for student filtering)
   - day_of_week (enum: Monday-Friday)
   - start_time, end_time
   - duration (minutes)
   - is_active

5. TeacherSession Schema:
   - timetable_id (ref: Timetable)
   - teacher_id (ref: User)
   - subject_id (ref: Subject)
   - room_id (ref: Room)
   - teacher_latitude, teacher_longitude
   - session_date
   - start_time, end_time
   - status (enum: scheduled, live, ended)
   - students_present, students_late, students_absent

6. Attendance Schema:
   - student_id (ref: User)
   - subject_id (ref: Subject)
   - teacher_id (ref: User)
   - session_id (ref: TeacherSession)
   - date
   - status (enum: present, late, absent)
   - time_marked
   - method (enum: auto, manual)
   - student_latitude, student_longitude
   - distance_from_room (meters)
   - distance_from_teacher (meters)
   - is_verified
   - notes
   - Unique index: student_id + subject_id + date

7. Notification Schema:
   - user_id (ref: User)
   - type (enum: success, warning, info, error)
   - title, message
   - related_model (attendance, class, etc.)
   - related_id
   - is_read
   - created_at

8. AttendancePercentage Schema (denormalized for performance):
   - student_id (ref: User)
   - subject_id (ref: Subject)
   - total_classes
   - present_count
   - late_count
   - absent_count
   - percentage
   - last_updated

========================
CRITICAL ATTENDANCE VALIDATION FLOW
========================
1. Student submits location during class time
2. Backend checks:
   - Is there an active TeacherSession for this time/subject?
   - Is student enrolled in this subject?
   - Is current time within class start-end window?
   - Is student within room geofence? (Haversine distance calculation)
   - Is student within teacher proximity range?
   - Has student already been marked today for this subject?
3. Calculate status:
   - If within first X minutes (e.g., 15 min): Present
   - If after X minutes but before class end: Late
   - If outside geofence: Reject
4. Create attendance record with all details
5. Update TeacherSession counters
6. Send notification to student
7. Update AttendancePercentage cache
